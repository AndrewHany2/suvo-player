## AI Slop Verdict
Mostly trustworthy and clearly hand-built on a real token system, not generic-cloned — a Linear/Stripe-fluent user would trust the shelves, posters, hero, and StatePanel. Two things make them pause. (1) The no-SVG constraint forces a View/glyph-drawn icon set with a few subtly-off symbols: `search` is ⌕ (U+2315 telephone-recorder), `signal` is ⦿ (also reused as the account-connected chip in the nav), and the icon-only `aspect` control is a bare rounded rectangle that reads as nothing without its a11y label. (2) The single biggest "wait, why did the UI just change?" moment is structural, not visual: an .mp4 opens the expo-video player (center 72px transport, stats/PiP/tune/info, buffered seek bar) while an .mkv opens the VLC player (bottom play/pause button, none of those, no buffered track) — same app, same task, two different chromes. Plus a permanent "HD" badge stamped on every poster regardless of actual quality reads as fake metadata. None of these are strangeness-without-purpose to the point of distrust, but they're the tells.

## Cognitive Load
Intrinsic load is well-managed on the browse surfaces (hero + horizontal rails + poster grid is the familiar model). Extraneous load spikes in the player: the expo-video bottom control row exposes up to 9 icon-only controls (speed, audio, cc, aspect, fullscreen, tune, info, timer, pip) that wrap on a phone — well past the >4-visible-options threshold, and several glyphs (aspect rectangle, tune sliders, info-i) aren't self-evident. Germane load is hurt by the mp4-vs-mkv player divergence: the user must relearn where play/pause lives per file type. Missing-state gaps: initial load shows a spinner instead of the existing skeletons; a failed category rail simply disappears with no 'couldn't load' affordance; History and LiveTV have no offline banner (only Movies/Series do), so on a dropped connection those two screens give no system status at all.

## Emotional Journey
Peaks are real: the Home hero on History with the 'Synced across your devices' line is a confident, cinematic entry, and the player's reconnect handling (translucent scrim over the retained last frame + 'Reconnecting…') is genuinely reassuring at the highest-stakes moment. Valleys cluster around setup and flaky network. First custom-channel add is the worst micro-journey — typing a URL blind under the keyboard, then a blocking Alert. On a flaky connection the browse screens quietly lose rails with no explanation, which reads as 'the app is broken' rather than 'the network hiccuped.' Peak-end is decent because playback errors end on a clear Retry/Close panel, but the two-players-one-task inconsistency erodes the sense that this is one polished product.

## Strengths
- Resilient-playback UX: the busy overlay is a translucent scrim over the retained last video frame (not an opaque panel), with distinct 'Loading…' vs 'Reconnecting…' copy and a fatal StatePanel offering Retry + Close — exactly the reassurance the flaky-network viewer needs (ExpoVideoPlayerScreen.native.jsx 785-825).
- Genuinely accessible, gesture-rich player: drag-to-seek/volume/brightness, double-tap ±10s, long-press 2x, PLUS an accessibilityRole='adjustable' seek bar with increment/decrement actions so VoiceOver/TalkBack users can scrub (ExpoVideoPlayerScreen 911-928, VlcPlayerScreen 619-641).
- Shared vocabulary is real, not aspirational: tokens.js + labels.js (frozen 'My List' / 'Continue Watching') + StatePanel + the shared PosterCard/ContentShelf give the browse surfaces one consistent, tokenized identity, and History's deferred-undo removal (useDeferredRemove + Undo toast) is a considerate touch (HistoryScreen.native.jsx 231-252).

## Priority Issues

### [P0] Menu/hint text fails WCAG AA contrast on surface2
- **File:** src/screens/ExpoVideoPlayerScreen.native.jsx:953
- **Why:** muted #7A86A8 on surface2 #1B2236 is ~4.2:1 — below the 4.5:1 body bar — and it's the color of every UNSELECTED row in the speed/audio/subtitle/sleep menus at 15px. Worse, the Add-Channel format hint is faint #4A5575 on surface2 at ~2.06:1, effectively invisible. This is the stated hard a11y bar, failing pervasively in the exact surfaces where the user makes playback choices.
- **Fix:** Use colors.text (#EAF0FF, ~13:1) for unselected menu rows and reserve muted for genuinely secondary/large text; raise the Add-Channel hint from faint to muted or text. Verify every muted/faint usage that sits on surface/surface2, not just on bg.
- **Command:** $impeccable contrast

### [P1] SkeletonPoster ignores prefers-reduced-motion
- **File:** src/presentation/components/SkeletonPoster.native.jsx:18
- **Why:** PosterCard correctly gates its shimmer on useReducedMotion, but SkeletonPoster runs its opacity loop unconditionally (no useReducedMotion import). Reduced-motion honored 'everywhere' is a stated hard bar, and every loading rail mounts a row of these pulsing placeholders.
- **Fix:** Import useReducedMotion in SkeletonPoster.native.jsx and skip the Animated.loop (render a static colors.surface block) when reduced motion or low-end, mirroring PosterCard.native.jsx lines 46-55.
- **Command:** $impeccable motion

### [P1] Add-Channel inputs are hidden behind the keyboard
- **File:** src/screens/LiveTVScreen.native.jsx:259
- **Why:** The Add Custom Channel sheet is anchored to the bottom (justifyContent flex-end) with two TextInputs, and there is no KeyboardAvoidingView anywhere. On iOS the software keyboard covers exactly the inputs the user is typing into — the first-time custom-channel setup is done blind.
- **Fix:** Wrap the sheet content in <KeyboardAvoidingView behavior={Platform.OS==='ios'?'padding':'height'}> (or move to a centered modal), and validate the URL with a scheme check before Alert.
- **Command:** $impeccable harden

### [P1] Initial screen load is a bare spinner, not the existing skeletons
- **File:** src/screens/MoviesScreen.native.jsx:37
- **Why:** The product bar is 'skeletons not spinners,' and the app already ships SkeletonPoster + a ContentShelf skeleton rail — but the top-level loading gate on Movies/Series/LiveTV (and CategoryGridPage) shows a plain centered Spinner via StatePanel. First paint teaches nothing and shifts layout when content swaps in.
- **Fix:** Render a header + 2-3 skeleton ContentShelf rows during the `loading` gate instead of StatePanel mode='loading'; reuse the skeleton path ContentShelf already has for items===null.
- **Command:** $impeccable states

### [P1] Two divergent player UIs for the same VOD task
- **File:** src/screens/VlcPlayerScreen.native.jsx:596
- **Why:** VideoPlayerScreen dispatches .mkv/.avi/etc to VlcPlayerScreen and everything else to ExpoVideoPlayerScreen. The expo player has a center 72px play/pause transport, a buffered seek track, stats/PiP/tune/info controls, and accessibilityRole/State on menu rows; the VLC player has none of these (bottom play/pause instead, no buffered indicator, no AUTH_EXPIRED error copy, bare menu rows). The user gets a different, less-capable UI purely based on container format.
- **Fix:** Extract the shared control chrome (transport, seek bar, menu row component with a11y state, error copy) into one component both engines render, so only the underlying <VideoView>/<VLCPlayer> differs.
- **Command:** $impeccable unify

### [P1] Failed category rails vanish silently; History/LiveTV have no offline state
- **File:** src/screens/LiveTVScreen.native.jsx:149
- **Why:** On a per-category fetch failure, loadChannelCategory sets the category to [] and ContentShelf returns null for empty items, so the rail disappears with no message or retry. Combined with the offline banner existing only on Movies/Series (not History/LiveTV), the flaky-network viewer sees content silently evaporate and can't tell if it's the network or the app.
- **Fix:** Distinguish 'failed' from 'empty' (e.g. items=false or an error flag) and render a compact 'Couldn't load — Tap to retry' rail; add the offline banner to History and LiveTV.
- **Command:** $impeccable states

### [P2] 'HD' badge stamped on every poster unconditionally
- **File:** src/presentation/components/PosterCard.native.jsx:91
- **Why:** PosterCard renders the HD badge for all cards whenever !LOW_END, independent of the item's real resolution. It's untrue metadata on most catalogs and adds constant visual noise, nudging toward the cluttered-IPTV anti-reference the brand explicitly avoids.
- **Fix:** Only render the HD badge when the item actually reports HD (e.g. a quality/resolution field), otherwise omit it.
- **Command:** $impeccable declutter

### [P2] Detail Back affordance reinvents the shared Button
- **File:** src/components/MovieDetail.jsx:107
- **Why:** MovieDetail and SeriesDetail hand-roll a rgba(0,0,0,0.55) back chip with web-only role/aria/tabIndex props, while CategoryGridPage uses the design-system <Button variant='ghost' icon='back'>. Same app, same gesture, two different back affordances — inconsistent vocabulary and duplicated a11y handling.
- **Fix:** Replace the hand-rolled chip in both detail screens with the shared ghost Button (or a single BackButton wrapper), dropping the web-only props on native.
- **Command:** $impeccable consistency

### [P2] Movies vs Series discover pills drift; Movies Downloaded pill has no press feedback
- **File:** src/screens/MoviesScreen.native.jsx:85
- **Why:** Sibling screens render the same pill row with different icon sizes (Movies size=14, Series size=16) and the Movies 'Downloaded' pill omits the pressStyle/hoverStyle the other pills have, so it gives no tap feedback while its neighbors do. Small, but it's exactly the subtle inconsistency that erodes 'one system' trust.
- **Fix:** Extract a DiscoverPill component shared by both screens with a single icon size and press feedback; delete the divergent inline copies.
- **Command:** $impeccable consistency

### [P2] Full-saturation indigo close button at rest
- **File:** src/screens/ExpoVideoPlayerScreen.native.jsx:842
- **Why:** Both players render the top-left close button with backgroundColor accentAlpha(0.9) — a near-solid indigo circle in a resting, non-active control. Indigo is the reserved 'active path' light and the brand bans heavy/full-saturation accents on inactive states; a persistent solid-indigo close reads louder than the content.
- **Fix:** Drop the close button to a neutral scrim (overlay/surface2 with a hairline) like the other transient controls, reserving indigo for genuinely active toggles.
- **Command:** $impeccable colorize

### [P2] Native Alert used for validation and success confirmation
- **File:** src/screens/LiveTVScreen.native.jsx:117
- **Why:** Add-Channel fires Alert.alert for both 'Missing Fields' and a 'Channel Added' success — a blocking OS modal for a routine confirmation is heavier than the moment warrants and clashes with the app's own toast/inline-error language (History already has an inline Undo toast).
- **Fix:** Show validation inline under the input and confirm success with the app's own transient toast pattern rather than a blocking Alert.
- **Command:** $impeccable states

### [P3] Minor consistency + hit-target nits
- **File:** src/presentation/components/ContentShelf.native.jsx:89
- **Why:** Shelf counts render raw ({count}) while the grid uses toLocaleString, so 12345 appears unformatted in rails; the seek track responder is only 26px tall (tap-to-seek < 44px target); VLC menu rows lack the accessibilityRole/State the expo rows carry; History's Undo target (~26px + hitSlop 8) is just under 44.
- **Fix:** Format counts in ContentShelf; grow the seek responder to 44px tall (visual bar can stay 4px); add accessibility props to VLC menu rows; bump Undo hitSlop.
- **Command:** $impeccable polish

## Persona Red Flags
- Viewer-on-flaky-network: hardest hit. Reconnect handling in the player is excellent, but on the browse side individual rails disappear with no message when a category fetch fails, and History/LiveTV never show the offline banner that Movies/Series do — so a network dip looks like a broken app rather than a transient hiccup.
- First-time playlist / custom-channel setup: the Add Custom Channel sheet puts its inputs under the iOS keyboard with no KeyboardAvoidingView, accepts any string as a stream URL with no format validation, and confirms via a blocking native Alert — a rough, error-prone first interaction.
- Cross-device continuity viewer: continuity messaging is a strength (History hero 'Synced across your devices'), but the promise of one consistent experience is undercut when the same 'play a movie' task yields two different player UIs depending on whether the file is .mp4 (expo) or .mkv (VLC).

## Minor Observations
- MoviesScreen initialNumToRender is 2+overscan while SeriesScreen hardcodes 3 — pick one derivation.
- discover-pill map leaves `idx` unused in both Movies and Series.
- VLC fatal-error screen has no AUTH_EXPIRED branch, so an auth failure shows the generic 'could not be played' while the expo screen explains it.
- gestureHint HUD uses fontWeight 700 on the body stack; fine, but it's the one transient element not pulling from the type ramp.
- CWCard, PosterCard, and the detail hero each independently re-declare the rgba(10,14,26,...) scrim stops instead of consuming tokens.scrim.native — a drift risk if the midnight value ever changes.
- expo player top bar shows two 'Ch' buttons (prev/next channel) distinguished only by a back vs chevron-right icon — easy to misread; consider 'Prev'/'Next' labels.
