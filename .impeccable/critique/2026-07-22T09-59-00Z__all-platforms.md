## AI Slop Verdict
Largely earns familiarity, with a few components that would make a Linear/Stripe-fluent user pause. The shared foundation is genuinely coherent: one tokens.js source of truth, one StatePanel for loading/error/empty, a frozen labels.js that killed the documented Favorites/My List/Watch History drift, and disciplined restraint (no shadows/glass, content-forward). The tells that read as unfinished rather than intentional: (1) a fabricated "HD" badge stamped on EVERY native poster regardless of actual stream quality — pure decorative filler that a careful user distrusts; (2) the TV see-all chevron rendered in Signal Cyan at rest, when cyan is documented as focus/hover-ONLY — a subtly-wrong accent that a design-fluent eye catches; (3) form placeholders too dim to read (2.1-2.6:1). None of this is pervasive slop; it is strangeness at the seams between surfaces, exactly where the "one account, every screen" promise is judged.

## Cognitive Load
Intrinsic load is low by design — content-forward, chrome recedes, no >4-option decision points in any shared component. Extraneous load creeps in at two points: (1) the always-on 'HD' badge on native posters is noise that carries no real information and competes with the rating badge in the same corner region; (2) sub-threshold placeholder text (faint 2.1-2.6:1) forces low-vision and even average users to guess what a field wants, converting a zero-effort hint into a decoding task — worst on the Auth/Accounts first-run forms. Germane load is well supported by skeleton posters (the placeholder footprint matches the real card, so nothing reflows). Missing-state gap: native Continue-Watching posters omit the resume progress bar + time readout that web/TV render, so the phone user loses the 'how far am I' signal the same account shows on desktop/TV.

## Emotional Journey
Peak: the cinematic Home hero (backdrop + display-font title + Play/Resume) and the instant, unambiguous cyan focus glow on posters/buttons — the app feels premium and responsive. Valley (high-stakes): the Auth and Add-Account forms, where placeholder hints sit at 2.1-2.6:1 and are effectively invisible — the one moment a new user must succeed, and the interface goes quiet exactly there. Secondary valley: a viewer who resumes a title on their phone sees no progress indicator, then opens the same account on desktop and sees one — a small 'did my progress even save?' doubt that the continuity promise is supposed to erase. Reassurance is present and good at the ends: empty states explain what will happen and that it syncs across devices; the error StatePanel gives a clear glyph + retry. Peak-end is decent; the mid-journey form valley is the fix-first moment.

## Strengths
- Single source of truth actually holds: tokens.js is consumed by JS primitives, native StyleSheet, and mirrored as literal hex in CSS; one StatePanel replaces every hand-rolled spinner/error/empty; labels.js is Object.frozen to prevent the documented per-platform copy drift from reappearing.
- Aurora restraint is real and consistent: depth from tonal layering (bg/surface/surface2), accent2 cyan reserved for interaction, focus glow gated on !isTV() with a literal box-shadow fallback — the design language survives the webOS file:// flattening as claimed.
- Button.web and Button.native are near-identical in prop API, size ramp, variant colors, and interaction model, and correctly use textStrong (white ~5:1) not text on the indigo primary fill — a deliberate, documented AA fix.
- Accessibility scaffolding is above the usual bar: role=status/aria-live=polite on loading, role=alert on error, real <button>/role=button + Enter/Space on poster cards, accessibilityRole/State on native buttons, 44x44 remove hit area.
- Skeletons (not spinners) reserve the real poster footprint on every platform's shelf, and the TV variant windows both axes off deterministic focus — polished loading behavior.

## Priority Issues

### [P0] Form placeholders use faint #4A5575 — 2.14-2.61:1, fails AA hard bar
- **File:** src/ui/tokens.js:24 (def); src/screens/AuthScreen.jsx:176,189; src/screens/AccountsScreen.jsx:210-226; src/screens/MoviesScreen.web.jsx:45; src/screens/SeriesScreen.web.jsx:279; src/screens/LiveTVScreen.web.jsx:652,723,739; src/screens/LiveTVScreen.native.jsx:224,263,264
- **Why:** faint on bg=2.61:1, on surface=2.34:1, on surface2=2.14:1 — all far below the 4.5:1 required for placeholder text (placeholders are NOT WCAG-exempt like disabled controls). This is the ONLY hint of expected input and it's used on the highest-stakes first-run surfaces: AuthScreen (email/password), AccountsScreen (host/username/password), Movies/Series/LiveTV search. A low-vision user cannot read what any field wants.
- **Fix:** Change placeholderTextColor from colors.faint to colors.muted (#7A86A8 = 4.77:1 on surface, passes) everywhere, matching what the TV build already does (AccountsScreen.tv.css:117 uses --a-muted). Reserve faint for genuinely disabled text only.
- **Command:** $impeccable harden

### [P1] Ghost Button label uses accent #6C5CE7 as text — 3.25-3.96:1, fails AA and violates Single-Light Rule
- **File:** src/ui/Button.web.jsx:41; src/ui/Button.native.jsx:37
- **Why:** accent as text = 3.96:1 on bg, 3.55:1 on surface, 3.25:1 on surface2 — below 4.5:1 for the normal-weight labels these buttons carry (Back at size sm=14px, Sign Out). It also breaks the Single-Light Rule: a resting, unfocused control is tinted with the active-path indigo when the rule says resting controls sit on the steel/slate ladder with no tint. The team already built accentText (#A99BF5 = 6.5-8:1) for exactly this and used it correctly in the TV letter-bar (--a-indigo-text), but the shared Button didn't adopt it.
- **Fix:** In variantStyle('ghost') set resting color to colors.text (or textDim), and promote to colors.accentText only on hover/focus (the active branch already tracks this). This fixes contrast AND the resting-tint violation in one change, on both platforms.
- **Command:** $impeccable colorize

### [P1] Resume progress absent on native posters — breaks continuity the same account shows on web/TV

- **Why:** PosterCard.web renders a watched/duration readout + accent progress bar for Continue-Watching cards (lines 146-153); TVPosterCard is a shim to the web card so TV gets it too. The same in-progress title therefore shows 'how far am I' on desktop and TV but nothing on the phone — directly undercutting the core 'continuity follows you to every device' promise for the most-used device.
- **Fix:** Port the watchedPct bar + fmtDur readout from PosterCard.web.jsx:69-153 into PosterCard.native (a 6px accent bar + overlay time chip), gated the same way on currentTime/duration so catalog cards are unaffected.
- **Command:** $impeccable adapt

### [P2] Fabricated 'HD' badge on every native poster — decorative, misleading, and native-only
- **File:** src/presentation/components/PosterCard.native.jsx:91-95
- **Why:** PosterCard.native unconditionally renders an 'HD' badge on all non-low-end cards (lines 91-95) with no basis in actual stream quality. It's a PRODUCT-banned decorative badge that conveys no real state, it can mislead (claims HD on SD content), and it appears on NEITHER web nor TV posters — so the same title looks different across the user's devices. The badge text also uses muted #7A86A8 at 9px, itself weak.
- **Fix:** Remove the HD badge entirely, or drive it from a real item quality field and add the identical badge to PosterCard.web so the vocabulary matches across surfaces. Removal is the lazy correct default.
- **Command:** $impeccable distill

### [P2] TV see-all chevron is Signal Cyan at rest — accent2 is documented focus/hover-ONLY
- **File:** src/presentation/components/VirtualShelves.tv.jsx:692
- **Why:** VirtualShelves.tv.jsx:692 renders the always-visible see-all chevron in colors.accent2 (cyan) at rest. The design rule reserves cyan strictly for focus/hover, never at rest (Single-Light). The equivalent affordance is colors.muted on web (ContentShelf.web.jsx:118) and native (ContentShelf.native.jsx:87), so this is both a rule violation and a cross-variant inconsistency: the same 'see all' cue is a quiet steel chevron on phone/desktop and a resting cyan beacon on TV.
- **Fix:** Change the TV see-all chevron color from colors.accent2 to colors.muted to match web/native and honor the accent2-is-interaction-only rule.
- **Command:** $impeccable colorize

### [P2] muted #7A86A8 fails AA as body text on surface2 (4.37:1)
- **File:** src/ui/tokens.js:23
- **Why:** muted computes to 5.32:1 on bg and 4.77:1 on surface (pass) but 4.37:1 on surface2 (#1B2236) — below 4.5:1. surface2 is the documented fill for chips, inputs, and elevated panels, and muted is the standard secondary-text color, so any secondary/label text on a chip or elevated card (e.g. counts, hints) is sub-AA. Also affects the native HD/badge text at 9px.
- **Fix:** For secondary text sitting on surface2, use textDim #B8C0DA (8.7:1) instead of muted; keep muted only on bg/surface. Alternatively lighten muted one step so it clears 4.5:1 on surface2.
- **Command:** $impeccable colorize

### [P2] danger #E5484D fails AA as small text on surface/surface2 (4.41 / 4.04:1)
- **File:** src/ui/tokens.js:25; src/screens/AccountsScreen.tv.css:74,84
- **Why:** danger = 4.92:1 on bg (pass) but 4.41:1 on surface and 4.04:1 on surface2 — below 4.5:1 for normal-size error text. StatePanel's error TITLE is safe (20px bold = large text, 3:1 bar), but inline/small danger text on elevated surfaces fails, e.g. AccountsScreen.tv.css:74 renders danger as both border and small button-label color on a surface.
- **Fix:** Use danger only for large/bold error text or as a border/icon color on surface/surface2; for small error body text on elevated surfaces, pair with bg background or lighten the danger token for text use.
- **Command:** $impeccable clarify

### [P3] Off-scale radii and an undocumented #fff literal in TV CSS

## Persona Red Flags
- Cross-device viewer (the core persona): resume progress visible on desktop/TV but missing on phone, and the same poster shows an HD badge only on native — the 'identical everywhere' promise visibly cracks between surfaces.
- Low-vision / AA-dependent user: cannot read form placeholders (2.1-2.6:1) on the login and add-account screens, and ghost 'Back'/'Sign Out' labels sit at 3.3-4:1 — blocked at exactly the setup steps that gate the whole app.
- Privacy-conscious user: well served — zero ads/trackers genuinely reads through in the restrained, content-first chrome; no dark patterns in the shared components.
- 10-foot TV user: focus order and D-pad model are first-class and the focus ring is unmistakable, but the resting-cyan see-all chevron trains the eye to read cyan as 'not-focused', muddying the focus signal cyan is supposed to own.

## Minor Observations
- StatePanel loading mode uses a centered Spinner while the rest of the app prefers skeletons; a full-screen skeleton (or at least the shelf skeleton row) would match the documented 'skeletons not spinners' preference.
- StatePanel default retryLabel='Retry' is a raw string literal rather than sourced from labels.js — small risk of future copy drift for a user-facing word.
- TV CSS relies on var(--a-*) throughout (e.g. MoviesScreen.tv.css letter-bar, LiveTVScreen.tv.css) while tokens.js justifies literal-hex on the premise that old webOS 'can't be assumed to support custom properties'; the two mental models contradict — reconcile the doc (the real floor is 71+ per project memory, so var() is fine and the literal-hex rationale can be softened).
- Native poster placeholder icon uses colors.faint (#4A5575) while web uses colors.muted — cosmetic, but another faint-vs-muted inconsistency for the same slot.
- HeroTV (Movies/Series featured) shows only a title — no meta, continuity label, or CTA — while Hero.web/Hero.native carry Play/Resume + secondary + meta; acceptable since it's the non-interactive featured billboard, but worth documenting so it isn't mistaken for a parity regression.
