# Anti-Patterns Verdict

**Does this look AI-generated?** Not broadly. A user fluent in the best media apps would trust most of Suvo. The token system (`src/ui/tokens.js`) is disciplined, `StatePanel` genuinely unifies loading/error/empty with correct ARIA, and the Aurora interaction language (cyan ring/glow on focus+hover only, hairline at rest, no resting shadow) is implemented with real rigor in `PosterCard.web.jsx`. `prefers-reduced-motion` is honored globally.

The slop tells are **localized, not systemic**, and they sit on load-bearing controls:
- The single most-clicked control — the Play/Continue button on every movie & series — is a white (`#fff`) black-text pill using literal `▶ ↺ ♥ ♡` glyphs (`MovieDetail.web.jsx:248-296`, mirrored in SeriesDetail). This is the exact "generic streaming clone" pattern DESIGN.md warns against and opts the signature control out of Aurora Indigo.
- Live cards carry an always-35% fake progress bar and a "Live · now playing" filler caption (`LiveTVScreen.web.jsx:146-168`).
- Off-palette greys (`#ccc`, `#aaa`) leak into the shell nav and dialogs.

**Deterministic scan:** detector over `src/screens src/components src/presentation src/navigation` → exit 2, **90 findings (87 advisory, 3 warning)**. Breakdown: design-system-color 56, design-system-font-size 20, design-system-radius 11, broken-image 2, overused-font 1. All 3 warnings are **false positives** (Inter is the documented body font; both `broken-image` hits are code comments, not markup). The 87 advisory findings are genuine token drift — literal colors/sizes/radii outside DESIGN.md — heaviest in the TV `.css` files and the player overlays. The color drift the detector caught (`#fff`×16, `#aaa`×3) is the same off-system palette the design review flagged on the play button and nav; the two assessments agree. Most player-overlay colors (`rgba(0,0,0,0.6)` scrims etc.) are legitimate-but-undocumented overlay values, not slop.

**Visual overlays:** No user-visible overlay available — browser automation was locked to a running Chrome instance for both assessments, so no injection was performed. Findings are grounded in source read + CLI detector.

# Overall Impression

The bones are excellent: a real design system, honest resilience (playback recovers silently instead of flashing "reconnecting"), and faithful Aurora interaction rules on the browse surface. What holds the web surface back is a small set of high-visibility inconsistencies — a signature Play button that looks like every other streaming app, sub-AA metadata text, and platform-broken feedback on the web target — plus a total absence of the product's headline promise (cross-device continuity) anywhere in the UI. The single biggest opportunity: make the most-used control (Play) and the app's whole reason to exist (continuity) *look* like Suvo.

# What's Working

- **`StatePanel.jsx`** — one component for loading/error/empty across all screens, with `role="status"`/`aria-live="polite"` and `role="alert"` correctly applied. "Resilience is felt" made concrete.
- **`PosterCard.web.jsx:100-159`** — faithful Aurora interaction rules: resting hairline, cyan ring + `GLOW_WEB` only on focus/hover, `box-sizing:border-box` so focus never shifts layout, full keyboard semantics.
- **`AuthScreen.jsx:50-74`** — excellent error recovery: network vs rate-limit vs unconfirmed-email vs bad-credentials each mapped to calm, specific, plain-language copy with a retryable network state.

# Priority Issues

**[P1] Primary Play CTA is off-system (white Netflix pill + emoji glyphs)**
- **What:** `MovieDetail.web.jsx:248-296` renders Play/Continue/From-Start/Favorites as `#fff` black-text pills using literal `▶ ↺ ♥ ♡` characters with white hover borders. SeriesDetail shares it.
- **Why it matters:** The most-clicked control in the app violates the Aurora Indigo primary rule, the Single-Light rule, and Icon-set consistency — the exact "generic streaming clone" anti-reference. Emoji glyphs also render inconsistently across OSes.
- **Fix:** Use the shared `Button` primary variant (Aurora Indigo `#6C5CE7` fill, Ice text) or the accent gradient for highest-emphasis Play; replace `▶/↺/♥` with `Icon name="play"/"refresh"/"star"`; drop white hover for `colors.accent2`.
- **Suggested command:** `$impeccable polish` (+ `$impeccable colorize` for the Single-Light reconciliation).

**[P1] Contrast failures: faint-steel used as visible metadata**
- **What:** `#4A5575` (documented placeholder/disabled ONLY) is used for readable copy — "Live · now playing" and channel counts (`LiveTVScreen.web.jsx:162,300`), "No channels found" (`:662`), shelf counts (`ContentShelf.web.jsx:120`). Faint on midnight ≈ 2.9:1, below the 4.5:1 AA hard bar.
- **Why it matters:** WCAG 2.1 AA is a stated hard bar. Sam (low-vision) can't read live metadata or empty-state guidance.
- **Fix:** Promote readable text to `colors.muted` (`#7A86A8`) or `colors.text`; reserve faint strictly for placeholder/disabled.
- **Suggested command:** `$impeccable harden`.

**[P1] Add-Channel feedback and validation are silent no-ops on web**
- **What:** `LiveTVScreen.web.jsx:467,495` call `Alert.alert` directly; `AccountsScreen.jsx:22-32` documents that react-native-web's `Alert` is a no-op stub. On the web/Electron/TV target, invalid submits show nothing and "Channel Added" never appears.
- **Why it matters:** Error prevention and status feedback both break on the primary platform. Riley submits an empty form and gets zero response; a successful add closes with no confirmation.
- **Fix:** Reuse AccountsScreen's `alertDialog`/`confirmDialog` web fallback (`globalThis.confirm`/`alert`) or an inline error row like AuthScreen's.
- **Suggested command:** `$impeccable harden`.

**[P2] Fake live progress + filler EPG**
- **What:** Every `LiveCard` renders a hardcoded `width="35%"` progress bar (`LiveTVScreen.web.jsx:154`) and a static "Live · now playing" caption; EPG line falls back to a bare `" "` (`:146`).
- **Why it matters:** A progress bar implies real program timing that doesn't exist — a dishonest affordance that erodes trust; also visual clutter.
- **Fix:** Drive the bar from real EPG now/next progress or drop it; drop the filler caption when no EPG is present.
- **Suggested command:** `$impeccable distill`.

**[P2] Video player control cluster: 8–10 icon-only options**
- **What:** `VideoPlayerScreen.web.jsx:663-843` — up to ten controls in one right-aligned row, all icon-only (`tune`/`settings`/`info` indistinguishable without hovering).
- **Why it matters:** Exceeds working-memory limits; Jordan (first-timer) and Casey (mobile, no hover) can't discover what half the icons do.
- **Fix:** Group secondary controls (stats, picture, sleep, cast) under the existing "more/tune" menu so the resting bar is ≤4–5 items; add short labels where space allows.
- **Suggested command:** `$impeccable clarify` / `$impeccable layout`.

**[P2] Banned side-stripe lives in a shared detail file**
- **What:** `MovieDetail.web.jsx:390-391` applies `borderLeftWidth={isTV ? 4 : 0}` in `colors.accent` — a 4px colored accent stripe, a named absolute ban. Gated to TV, but sits in the shared component.
- **Why it matters:** Direct violation of the ban list and a latent risk if the gate is removed.
- **Fix:** Replace with tonal elevation (`surface2` panel) instead of a colored left border.
- **Suggested command:** `$impeccable polish`.

# Persona Red Flags

**Sam (accessibility-dependent):** faint-steel `#4A5575` (~2.9:1) on "Live · now playing," channel counts, "No channels found," shelf counts (`LiveTVScreen.web.jsx:162,300,662`; `ContentShelf.web.jsx:120`) — below AA. The 10-icon player row is icon-only with tooltip-only labels; a sighted low-vision user gets no text. The fake live progress bar has no accessible meaning.

**Casey (distracted mobile):** Play and Back sit at the **top** of `MovieDetail` (`:126-149,244`), outside the thumb zone; no bottom action bar. Hover-only affordances (shelf nav arrows on `:hover`, `AppNavigator.web.jsx:109`; CW play-overlay on `:hover`, `:128`) don't exist on touch, so those controls are invisible on a phone. The 10-icon player cluster is tap-hostile at phone width.

**Riley (stress tester):** Empty Add-Channel submit on web produces nothing (Alert no-op, `LiveTVScreen.web.jsx:467`). A channel adds silently with no confirmation. A brand-new connected account lands on empty Home ("Nothing here yet", `HistoryScreen.web.jsx:387-396`) whose only escape is the top nav — no browse CTA on the landing screen. No stream-URL format validation.

**Morgan (project-specific — the cross-device continuity viewer):** The headline promise is "one account, every screen — history, My List, resume follow you." Yet there is **no visible sync status anywhere** — no "synced" indicator, no last-device hint, no reassurance the couch-to-train handoff worked. `goToTab` silently refetches (`AppNavigator.web.jsx:418-429`) but the user never sees continuity is live. On a second device with empty resume state, Morgan can't tell "still syncing" from "nothing saved." The strongest brand differentiator is completely unexpressed in the UI.

# Minor Observations

- Boot tab is `"live"` (`AppNavigator.web.jsx:374`) while the first nav item is "Home" — the app opens on the second concept.
- Inactive nav label hardcoded `#ccc` (`AppNavigator.web.jsx:199`); dialog close + exit-prompt subtitle use `#aaa` (`:745,806,862`) — off the steel/slate ladder.
- Player seek fill and volume use cyan `accent2` as a **resting** fill (`VideoPlayerScreen.web.jsx:160-168,178`); per Single-Light, the active playback path should be indigo (`accent`), cyan reserved for focus.
- The `age` badge uses `colors.accent` border+text at rest (`MovieDetail.web.jsx:224-240`) — indigo as decorative resting tint, a soft Single-Light violation.
- Two near-duplicate CategoryPage implementations (Movies vs Series) differ only in card component and gap — drift risk.
- 87 advisory token-drift findings (colors/sizes/radii) — mostly TV `.css` literals and player overlay scrims; real drift, low individual severity, worth a consolidation pass into tokens.
